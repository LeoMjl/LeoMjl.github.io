# Agent里最容易被忽视的坑：你的while循环写对了吗？

你有没有遇到过这样的场景：你给Agent布置了一个任务——帮我查一下下周去上海的机票。模型思考了10秒，调用了机票查询工具，又思考了10秒，再调了一次，再思考，再调用……循环了十几轮，最后没订成票，反而花掉了几毛钱的API费用，外加一堆毫无意义的日志。你第一反应是不是："这个模型也太笨了"？

其实，**大部分时候，问题真的不在模型**。模型只是那个做出决策的"大脑"，而真正决定大脑如何一步步工作、什么时候停、什么时候重试、什么时候换策略的，是包裹在模型外面的那个"执行循环"——也就是Agent Loop。它看起来简单得离谱，几乎每个人写第一版Agent代码时都写过它：`while True: think_and_act()`。但就是这个"看起来很简单"的循环，决定了一个Agent是聪明、稳定、省Token，还是愚蠢、疯狂、烧钱。

## 一、四种Agent Loop范式，你用到了第几层？

如果把Agent Loop的演进看作一条路，开发者通常会经过四个阶段。每个阶段都对应一种典型的代码模式。

**阶段一：朴素while循环**

最原始的写法，就是一个无限循环，每轮让模型看一下当前状态，做一次行动。代码大概长这样：

```python
while True:
    user_input = get_user_query()
    thought = model.think(user_input, history)
    action = choose_action(thought)
    result = execute_tool(action)
    history.append(f"执行结果：{result}")
```

这种写法的好处是零门槛，几分钟就能搭出一个能跑起来的Agent。**但问题也就在这里**：它没有任何结构。模型不知道什么时候该停，什么时候该问用户，什么时候该换一个工具。任务稍微变复杂，它就会陷入"思考——行动——再思考——再行动"的无底洞。

你可以把它想象成一个完全没有菜谱的厨师：每次做完一道菜，才去问老板"接下来做什么"，老板说"你看着办"，于是厨师可能把盐加了三次，或者一直重复切菜。

**阶段二：ReAct（Reasoning + Acting）——给循环装上导航**

为了解决"无脑循环"，ReAct模式被广泛采用。它在每步行动之前强制要求模型先输出一个"思考"，然后再选择行动，接着观察行动结果，形成"Thought-Action-Observation"三步。

```python
while not task_done:
    thought = model.reason(observation, history)
    action, tool_input = model.choose_action(thought)
    observation = execute_tool(action, tool_input)
    if "最终答案" in thought:
        break
```

ReAct循环相当于给厨师配了一个对讲机：每做一道菜，他都要先告诉老板"我接下来想做这个，因为我觉得这样更符合客人口味"，然后再动手，做完再汇报结果。这样一来，行为变得可追踪，模型也不那么容易跑偏。

**阶段三：Plan-Execute——先把地图画出来再走**

ReAct虽然减少了无意义循环，但仍然是一步一抬头，遇到需要七步才能完成的任务时，它很容易在第四步迷路，因为每步都只看眼前。Plan-Execute模式的做法是：**接到任务后，先花一轮时间生成完整的执行计划（Plan），再逐步执行**。

```python
plan = model.create_plan(goal)
for step in plan:
    observation = execute_step(step)
    if should_replan(observation):
        plan = model.replan(goal, completed_steps)
        # 重新进入新计划
```

这就像厨师拿到一桌宴席的菜单后，先把所有菜的做法和顺序列出来，再按计划一道道做，中途发现食材不够时才调整。这种模式非常适合需要多步操作、且有明确终态的任务，比如"分析这份数据并生成PPT"。

**阶段四：Harness引擎（声明式调度）——你不用写while了**

前面三种范式，本质上都是开发者自己在写循环逻辑——while、for、if判断状态转移。但到了生产环境中，任务分支越来越复杂，要处理的异常情况（工具挂了、模型返回格式错误、用户中途修改要求……）指数级增加，手写的while循环会变成一个不可维护的"状态怪物"。

于是，新一代框架（如LangGraph、Dify的工作流引擎）开始提供**声明式循环**：开发者不再写"while什么条件下怎么做"，而是定义一组"节点"和节点之间的"边"，由引擎自动管理整个执行流。

```python
from langgraph import StateGraph

graph = StateGraph(AgentState)
graph.add_node("agent", call_model)
graph.add_node("tools", execute_tools)
graph.add_conditional_edges(
    "agent",
    should_continue,
    {"continue": "tools", "end": END}
)
graph.add_edge("tools", "agent")
app = graph.compile()
result = app.invoke({"input": user_query})
```

你只是声明了"agent节点后面可以走到tools节点，也可以结束"，而**循环本身是框架在背后帮你跑的**。就像把整个餐厅的运营流程写进了一套自动化系统，厨师、传菜、结账的协作不需要一个人站在中间喊"下一步该谁了"。

这四种范式不是"哪个最先进就用哪个"，而是对应不同复杂度阶段的合理选择。下一节我们会看到，很多项目的崩溃，恰恰因为在不合适的阶段用了不合适的循环。

## 二、为什么大多数Agent项目，会卡在最不起眼的Loop设计上

Loop设计一旦出错，后果往往不是报个错就完事，而是系统"慢性死亡"。

**Token暴增，成本悄然失控**

在while循环中，很多开发者习惯把每次工具结果原封不动地堆进上下文历史。一个简单的"帮我查三只股票"任务，模型先查第一只，工具返回了2000字的网页原文，第二只又是2000字，第三只再2000字……到第四轮，上下文窗口里已经塞了6000字几乎无用的数据，但模型还得一字不漏地"看"完再输出思考。结果就是：一次任务消耗的Token，可能抵得上普通对话一百轮。

**状态"失忆"，越跑越偏**

当你用Plan-Execute模式时，模型生成的计划存进某个变量，但在执行第三步时，因为上下文窗口被剪裁，前两步的观察结果丢失了。模型发现"咦，我前面做了什么来着？"，于是重新规划，甚至从头开始。这种隐形状态丢失，常常被误认为模型出现了幻觉。

**死循环那些"聪明"的伪装**

最隐蔽的死循环不是`while True`里永远不走`break`那种，而是模型**看起来每次都在做不同的事，但始终没在向目标靠近**。比如让它优化一份代码，它反复运行测试、微改一行、再测试，循环十次后测试依然报错。每次输出都不同，但任务没半点推进。因为循环里缺少一个收敛条件。

**工具调用混乱：选择太多等于没有选择**

当Agent连接了20个工具，在ReAct循环里每步都要从20个工具里挑一个时，模型就像一个站在50种调料前的厨师，容易选错，选错后观察到错误结果，再思考，再选错，最终无法收束。如果你没有在Loop里做工具分层或消歧逻辑，这种"选择瘫痪"几乎必然发生。

## 三、一张图不够？这套选型框架直接帮你做决定

面对四种范式，没有银弹，但有一套非常实用的判断标准。

- **任务是否单轮就能完成？**
  如果只是翻译、回答一个明确问题，且不需要调用工具，那连while都不用写，单次模型调用就解决了。如果需要用工具但工具结果就是最终答案（比如"查天气"），朴素while循环足够，但记得加一个最大轮次限制。

- **工具结果需要被理解、再决策？**
  这是ReAct的用武之地。典型场景如客服Agent：用户说"我要退货"，模型需要先查订单、再判断条件、再生成退货单，每一步的观察都会影响下一步。ReAct天然的"思考-行动-观察"循环很契合。

- **任务有明确的分解步骤，且需要保证完整性？**
  比如"爬取一个网站的所有文章并生成摘要"，或者"分析Excel数据并输出PPT"。这种任务用Plan-Execute能减少翻来覆去的尝试，先花几十个Token生成计划，远好过上百次无效工具调用。

- **多人或长时间运行的复杂工作流？**
  例如企业内部的审批流Agent、多Agent协作的软件开发管线。这种情况下，手写while去管理所有分支、异常回滚、人机交互点简直是一场灾难。使用Harness引擎（LangGraph、Temporal等）让循环成为基础设施的一部分，你只需关注业务节点。

一个简单的心法：**能用单次调用就不用循环，能用ReAct就避免Plan-Execute，只有当任务本身的结构感大于探索感时，才上Plan-Execute，而一旦系统里有超过两个条件的环路分支，立刻迁移到声明式引擎。**

## 四、从"写循环"到"设计循环"——Agent工程师的思维跃迁

大多数开发者开始写Agent时，是把模型当成一个"更聪明的函数"，把循环当成"调用这个函数的东西"。但真正的Agent工程，需要你**把循环本身当作产品去设计**。

**把状态提到一等公民**
别让状态散落在`history`列表和各种临时变量里。定义一个明确的State对象，里面有消息历史、计划步骤、已执行的步骤、用户原始目标。循环的每个节点都从State读、也向State写。这样不仅逻辑清晰，调试时也能随时看到Agent"当前站在哪个位置"。

**为循环写测试，而不是只测模型**
你可以用模拟的工具和预定义的模型输出来驱动整个循环，验证在给定状态下Agent是否走了正确的边、是否在适当的时候退出。一个没有循环测试的Agent项目，上线后第一个异常场景就会原地爆炸。

**观测你的循环**
在循环的每个出入口打上日志：目前是第几轮、当前使用的工具、Token累计量、这一步的思考摘要。这些数据会告诉你：是循环跑偏了，还是模型变弱了。没有观测的Loop就是开夜车没开大灯。

**拥抱声明式，但别过早**
声明式循环很强大，但也有学习成本。如果你的任务目前用10行while就能搞定并且稳定运行，没必要硬上LangGraph。但当代码里开始出现`if task_type == 'A' and not retried and ...`这种意大利面条，请毫不犹豫地重构为声明式图——那是在帮你，而不是给你添麻烦。

## 五、总结：Loop是Agent的骨骼

很多团队把90%的精力花在选模型、调Prompt、造工具上，却只在最后花10分钟写了个`while True`把东西串起来。这就像精心打磨了发动机、变速箱、内饰，最后却用一根麻绳来传动。**Agent Loop决定了一个Agent的行动模式、资源消耗边界和错误恢复能力**。

下次当你看到Agent像一只无头苍蝇一样重复调用同一个API时，不要马上去换一个更强的模型。打开那个被你忽视了几十次的主循环，理一理它的退出条件、状态传递和工具选择逻辑。你会发现，很多时候只需要重构一个循环，就能让一个"人工智障"瞬间变回你想象中的智能助手。
