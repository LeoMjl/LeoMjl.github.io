# AI Agent多角色协作：从脚本到乐团的架构升级

![AI Agent 多角色协作封面图](../assets/cover.png)

2025年以来，AI Agent从一个"会调API的聊天机器人"变成了能独立完成任务的工作单元。但单个Agent很快碰到了天花板——一个Agent再聪明，也无法同时做好调研、写作、配图、审核和发布。就像一个全栈工程师能搞定小项目，但做一个完整产品需要设计师、前端、后端、测试和运维各司其职。

多角色协作架构，正是解决这个瓶颈的答案。本文将用 Hermes Agent 的实战架构为例，拆解多智能体协作的核心设计模式。

## 一、单Agent的天花板：为什么一个聪明的AI不够用

单个LLM驱动的Agent看似万能——它能写代码、能查资料、能生成图片。但当你试图让它完成一个完整的博客发布流程（选题→调研→写作→配图→审核→发布），问题就来了。

**上下文窗口被撑爆。** 一个任务涉及的数据越多，Agent的记忆负担越重。调研阶段读过20篇网页后，写作阶段能记住的只有前3篇。就像让一个人同时记住所有原材料、菜谱步骤和摆盘要求——大脑会溢出。

**角色冲突无法解决。** 同一个LLM既要当"创意写手"写出吸睛标题，又要当"严格编辑"挑出逻辑漏洞。这两个角色天然冲突——前者追求惊艳，后者追求严谨。在同一个提示词里让模型同时扮演这两个角色，结果往往是两头不讨好。

**线性流程缺乏容错。** 传统自动化是「A→B→C→D」的流水线：完成A才能到B。当A出错时，整个流程卡死。而真实工作中的创造过程是非线性的——编辑发现配图有问题，需要回到配图阶段重做，而不是整个流程从头跑。

这三个瓶颈指向同一个结论：**真正复杂的任务，需要多个Agent像团队成员一样各司其职、互相交付、并行推进。**

## 二、多角色协作的三层架构：从脚本到乐团的跃迁

![线性管道 vs 多角色编排架构对比](../assets/architecture-comparison.png)

如果说单Agent是一条传送带上的机械臂，那多Agent协作就是一支交响乐团。每个乐手有自己擅长的乐器（角色），指挥负责协调节奏和信息传递（编排器），而总谱定义了每个声部何时进入、何时休止（工作流）。

在这个类比下，多Agent系统的三层架构是：

**第一层：角色定义。** 每个Agent被赋予明确的角色——研究员、写手、配图师、编辑、发布员。这不仅是不同的提示词，更是不同的"人格"：研究员追求事实准确，写手追求表达吸引力，编辑追求结构完整。在 Hermes Agent 中，每个角色的行为规范定义在独立的 `SOUL.md` 文件中——研究员不会替用户做选题决策，编辑不会凭空通过带占位符的草稿。

**第二层：编排逻辑。** 编排器（Orchestrator）不是另一个Agent，而是一组管理状态的规则。它决定什么条件下研究员该重跑、什么输出算"编辑通过"、什么信号触发配图返工。好的编排不是 if-else 嵌套，而是声明式的状态转换——

- 研究员产出 `topic-brief.md` → 选题确认 → 启动写手
- 写手产出草稿 → 配图师嵌入图片 → 编辑审核
- 编辑返回 `revision_needed` → 路由到对应角色重做 → 下游角色级联重跑

**第三层：工具集成。** 每个Agent不是凭空推理——它们共享一套工具链。浏览器工具、文件系统、图片生成API、MCP服务器。关键设计原则是：工具属于系统而非某个Agent。所有角色都能读写同一个目录、调用同一套API，只是权限和作用域不同。

## 三、Skill系统：让Agent拥有"专业技能"

![Skill 模块化架构示意图](../assets/skill-system.png)

如果说角色定义了"谁来做"，那Skill定义了"怎么做"。Skill是Agent能力的可复用单元——一个Skill包含触发条件、执行步骤、工具调用、边界约束和失败回退。

以 Hermes Agent 的博客协作 Skill 为例，它不是一段"告诉Agent写一篇博客"的提示词，而是一个完整的操作手册：

- **触发条件：** 用户说"写博客""blog""选题到发布"
- **前置检查：** 确认输出目录、读取 AGENTS.md
- **执行步骤：** 13步状态机，从 researcher → writer → illustrator → editor → 草稿审核 → 平台选择 → publish
- **失败处理：** 浏览器阻断→回退到模型推理；配图生成失败→标记缺失并路由返工；终端不可用→切换到 execute_code 直调 API
- **硬约束：** 标题必须 ≤30字符且用 `len()` 程序化验证；编辑审核前必须完成图片嵌入；存在 `配图待生成` 占位符的草稿禁止通过

这就是Skill和普通提示词的本质区别：**Skill是工程化的操作规范，不是自然语言建议。** 它定义了确定的输入输出、失败模式和恢复路径。一个Agent加载了Skill之后，完成任务的可靠性从"大概率"变成了"可预期"。

多Agent系统中，Skill的另一个关键作用是**组合复用**。博客流程需要"写手"和"编辑"，代码审查流程也需要"写手"和"编辑"——同一个Skill可以在不同编排场景中复用。当你积累了10个Skill时，你不是多了10条提示词，而是多了一套可以任意组合的能力矩阵。

## 四、MCP协议：Agent的"手和脚"

再聪明的Agent，如果只能"想"不能"做"，就是一个困在聊天框里的灵魂。Model Context Protocol (MCP) 解决了这个问题——它给Agent接上了手和脚。

MCP的核心思想很简单：**把外部能力抽象成标准化的工具接口。** 就像USB协议让任何设备都能插到电脑上一样，MCP让任何服务都能被Agent调用。

在 Hermes Agent 的架构中，MCP服务器承担了三类角色：

- **浏览器控制：** 今日头条MCP通过Chrome DevTools Protocol操控真实浏览器——登录、填写标题、上传图片、点击发布。不是模拟HTTP请求，而是像真人一样操作网页。这意味着它能应对验证码（需要用户介入）、处理动态加载的内容、感知真实的发布状态。
- **平台集成：** 知乎MCP通过Draft.js编辑器的键盘级模拟逐字输入内容，因为React的Draft.js会忽略DOM注入——只有真实的键盘事件才能触发React状态更新。这种级别的工程细节，是一般"调用API发文章"做不到的。
- **本地工具链：** 文件系统读写、Python脚本执行、Qwen Image生图API调用——所有这些能力通过统一的MCP接口暴露，Agent无需知道底层实现。

架构上的关键洞察：**MCP服务器不是Agent的一部分，而是Agent之外独立运行的进程。** 这意味着：
1. 服务器崩溃不影响Agent主进程
2. 服务器可以独立重启、升级、调试
3. 多个Agent可以共享同一个MCP服务器（同一个Chrome实例被多个Agent复用）

## 五、从0到1：如何在你的项目中引入多Agent协作

![多Agent系统搭建路径图](../assets/roadmap.png)

多Agent架构不是只有大团队才需要的东西。即使是个人项目，引入角色分离也能显著提升Agent的可靠性。以下是渐进式引入路径：

**第一阶段：角色分离。** 把你现在的"全能Agent提示词"拆成两个角色——一个负责"产出"（写手），一个负责"审核"（编辑）。写手输出草稿，编辑拿到草稿后独立判断是否需要修改。你会立刻发现：让两个角色"互相检查"比让一个角色"自我审查"效果好得多。

**第二阶段：引入编排规则。** 用明确的文件作为状态载体——编辑审核通过的标记不是"回答里说通过了"，而是一个真实的 `editor-review.md` 文件，里面有结构化的 YAML 状态块。这让你的编排逻辑变得可追溯、可调试、可自动化。

**第三阶段：接入MCP工具。** 找到你工作流中需要"人类操作"的环节——发布到平台、操作浏览器、处理文件——选一个接入MCP。原则是"先接一个、跑通再扩"。一个能自动发布文章的Agent，比一个什么都能做但什么都做不好的Agent，价值高得多。

**第四阶段：建立Skill库。** 每当你完成一个稳定的工作流，把它固化为Skill。Skill应该包含：触发条件、输入格式、输出路径、失败处理。好的Skill像一个函数——你不需要记住它的内部实现，只需要知道"给定X，产出Y，如果出错回退到Z"。

---

多Agent协作不是为了让架构图看起来更复杂，而是为了解决一个根本问题：**单个大脑有认知上限，但协作系统没有。** 当你的Agent不再是一个人包揽所有环节，而是一支各司其职的团队时，它才能真正完成那些需要多步决策、多角色参与、多工具协同的复杂任务。
